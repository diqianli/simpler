深入理解动态任务图的构建过程 - TensorMap 查找、依赖链构建、性能热点

1. pto2_submit_task 执行流程

核心函数：任务提交与依赖发现

每次调用 pto2_submit_task() 都会执行以下 8 个步骤，动态构建任务图：

0 sync_tensormap() O(m)
同步 TensorMap，清理过期条目
1 task_ring.alloc() O(1)
从任务环分配槽位，可能阻塞
2 TensorMap.lookup() O(n)
查找 INPUT 参数的依赖，遍历哈希链
3 去重检测 O(n²)
检查 producer 是否已在 fanin 列表中
4 fanout_lock() 阻塞
获取生产者的自旋锁，添加消费者
5 heap_ring.alloc() O(1)
分配输出缓冲区，可能阻塞
6 TensorMap.insert() O(1)
注册 OUTPUT 到 TensorMap
7 scheduler.init_task() O(1)
初始化调度器状态，检查是否 READY

🔥 性能热点 (Step 2, 3, 4)

这三个步骤在高并发或复杂依赖场景下会成为瓶颈：

  • Step 2: 哈希链过长时遍历开销大
  • Step 3: 线性搜索去重，O(n²) 复杂度
  • Step 4: 自旋锁竞争，多消费者同时访问同一生产者

3. TensorMap 查找机制详解

lookup() 函数执行流程

当提交任务时，对每个 INPUT 参数调用 lookup() 查找依赖：

// pto_tensormap.h: lookup() 函数 void lookup(const Tensor& query, PTO2LookupResult& result) { result.count = 0; // Step 1: 计算哈希值 uint32_t bucket_idx = hash(query.data().buffer.addr); // Step 2: 获取桶的链表头 PTO2TensorMapEntry* cur = buckets[bucket_idx]; PTO2TensorMapEntry** prev_ptr = &buckets[bucket_idx]; // Step 3: 遍历链表 while (cur != nullptr) { // Step 3a: 检查条目是否过期 if (!entry_valid(*cur)) { // 惰性清理：截断过期条目 *prev_ptr = nullptr; while (cur) { remove_entry(*cur); cur = cur->next_in_bucket; } return; } // Step 3b: 检查地址匹配 if (query.data().buffer.addr == cur->addr) { // Step 3c: 检查内存重叠 OverlapStatus status = query.is_overlap(cur->tensor.data()); if (status != OverlapStatus::NO_OVERLAP) { result.push(cur, status); // 找到依赖! } } prev_ptr = &cur->next_in_bucket; cur = cur->next_in_bucket; } }

⚠️ 性能问题

  • 链表遍历: 最坏情况 O(n)，n = 桶中条目数
  • 哈希冲突: 同一地址的多个 view/reshape 会落入同一桶
  • 重叠检测: HBB 计算可能较复杂
// pto_tensormap.h: insert() 函数 void insert(const Tensor& tensor, int32_t producer_id, bool with_alloc) { // Step 1: 分配条目 PTO2TensorMapEntry* entry = new_entry(); // Step 2: 初始化条目 entry->tensor = tensor; entry->producer_task_id = producer_id; entry->with_alloc = with_alloc; entry->addr = tensor.data().buffer.addr; // Step 3: 插入到桶头部 (O(1)) entry->bucket_index = hash(entry->addr); entry->next_in_bucket = buckets[entry->bucket_index]; entry->prev_in_bucket = nullptr; if (entry->next_in_bucket) { entry->next_in_bucket->prev_in_bucket = entry; } buckets[entry->bucket_index] = entry; // Step 4: 链接到任务的条目列表 int32_t slot = producer_id & (TASK_WINDOW_SIZE - 1); entry->next_in_task = task_entry_head[slot]; if (entry->next_in_task) { entry->next_in_task->prev_in_task = entry; } task_entry_head[slot] = entry; }

内存重叠检测

TensorMap 不仅检测精确匹配，还检测内存区域重叠：

PARTIAL_OVERLAP (部分重叠)
A
B

B 部分覆盖 A，B 依赖 A

COVERED (完全覆盖)
A
B

B 完全覆盖 A，可移除 A 的条目 (INOUT 优化)

4. 依赖链构建详解

// pto_orchestrator.cpp: submit_task 中的依赖构建 // 1. 临时数组收集 fanin int32_t fanin_temp[PTO2_MAX_INPUTS]; int32_t fanin_count = 0; // 2. 遍历 INPUT 参数 for (int i = 0; i < num_params; i++) { if (params[i].type == INPUT || params[i].type == INOUT) { // 查找 TensorMap PTO2LookupResult result; tensor_map.lookup(params[i].tensor, result); for (int r = 0; r < result.count; r++) { int32_t producer_id = result.entries[r].entry->producer_task_id; // 🔴 O(n²) 去重检测! bool already_added = false; for (int j = 0; j < fanin_count; j++) { if (fanin_temp[j] == producer_id) { already_added = true; break; } } if (!already_added) { fanin_temp[fanin_count++] = producer_id; // 添加到 producer 的 fanout 列表 pto2_add_consumer_to_producer(orch, producer, producer_id, task_id); } } } } // 3. 构建 fanin 链表 for (int i = 0; i < fanin_count; i++) { task->fanin_head = pto2_dep_list_prepend(&dep_pool, task->fanin_head, fanin_temp[i]); } task->fanin_count = fanin_count;

🔴 O(n²) 去重检测

当任务有大量输入时（如 concat 100 个张量），去重检测成为瓶颈。

优化建议: 使用位图或哈希集合，将复杂度降到 O(1)

5. fanout 自旋锁分析

// pto_runtime2_types.h: fanout 锁 static inline void pto2_fanout_lock(PTO2TaskDescriptor* task) { while (PTO2_EXCHANGE(&task->fanout_lock, 1) != 0) { PTO2_SPIN_PAUSE_LIGHT(); // 自旋等待! } } // pto_orchestrator.cpp: 添加消费者 void pto2_add_consumer_to_producer(...) { pto2_fanout_lock(producer); // 获取锁 // 添加消费者到 fanout 链表头部 producer->fanout_head = pto2_dep_list_prepend(...); producer->fanout_count++; // 检查生产者是否已完成 (early-return) if (producer_state >= COMPLETED) { __atomic_fetch_add(&consumer->fanin_refcount, 1, SEQ_CST); } pto2_fanout_unlock(producer); // 释放锁 }

🔴 锁竞争场景

当多个消费者同时依赖同一生产者时（broadcast 场景），自旋锁导致严重竞争：

  • 线程 1, 2, 3 同时尝试添加到 Task A 的 fanout 列表
  • 只有 1 个线程能获得锁，其他线程自旋等待
  • 在 AICPU 并行模式下尤为严重
🔴 有锁竞争 (当前实现)
🟢 无锁优化 (建议)
