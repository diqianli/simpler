2. 关键瓶颈详解

🔴 重复检测线性搜索 O(n²)
pto_orchestrator.cpp:349-354
检查 producer 是否已在 fanin 列表中时，使用线性搜索数组。
// 对每个新 producer，遍历已有列表 for (int j = 0; j < fanin_count; j++) { if (fanin_temp[j] == producer_task_id) { already_added = true; break; } }
触发场景: 任务有大量输入参数 (如 concat 100 个张量)
优化: 使用位图或哈希表去重，O(1) 查找
🔴 TensorMap 链表遍历 O(n)
pto_tensormap.h:208-247
哈希冲突导致链表过长，每次 lookup 需要遍历整个链。
while (cur_entry != nullptr) { // 检查每个条目... if (!entry_valid(*cur_entry)) { // 遇到过期条目才截断 break; } // 重叠检测... cur_entry = cur_entry->next_in_bucket; }
触发场景: 大量张量使用同一基地址 (view/reshape 操作)
优化: 增加桶数量 / 更激进的链截断 / 开放寻址
🔴 fanout 自旋锁竞争 阻塞
pto_orchestrator.cpp:207, pto_runtime2_types.h:366
多个消费者同时依赖同一生产者时，自旋锁导致线程阻塞。
// 获取自旋锁 static inline void pto2_fanout_lock(PTO2TaskDescriptor* task) { while (PTO2_EXCHANGE(&task->fanout_lock, 1) != 0) { PTO2_SPIN_PAUSE_LIGHT(); // 自旋等待! } }
触发场景: 高并发，多个任务同时读取同一输入 (broadcast)
优化: 无锁队列 / 细粒度锁 / 批量添加
🟡 频繁原子操作 开销
pto_orchestrator.cpp:216-222, 240-241
高并发下原子操作成为热点，缓存行争用严重。
// 每次添加消费者都要原子操作 __atomic_load_n(&orch->aicpu_task_completed[prod_slot], __ATOMIC_ACQUIRE); __atomic_fetch_add(&sched->fanin_refcount[cons_slot], 1, __ATOMIC_SEQ_CST);
触发场景: 多线程并发提交任务 (AICPU 并行模式)
优化: 批量更新 / thread-local 缓存 / relaxed 内存序
🟡 堆分配阻塞 阻塞
pto_ring_buffer.h:87-132
内存不足时自旋等待，无法做其他工作。
while (1) { void* ptr = pto2_heap_ring_try_alloc(size); if (ptr != NULL) return ptr; // 空间不足，自旋等待... PTO2_SPIN_PAUSE(); }
触发场景: 任务窗口小，生产快于消费
优化: 异步通知 / 动态窗口调整 / 预分配
🔵 TensorMap 同步开销 O(m)
pto_tensormap.cpp:236-250
每次提交都同步，可能触发昂贵的清理操作。
void sync_tensormap() { int32_t new_last_task_alive = PTO2_LOAD_ACQUIRE(...); sync_validity(new_last_task_alive); if (需要清理) { cleanup_retired(...); // O(m) 操作! } }
触发场景: 周期性清理触发时
优化: 延迟清理 / 异步清理 / 增量清理

5. 优化建议汇总

针对各瓶颈的优化策略

1. O(n²) 去重优化
  • 使用位图记录已添加的 producer (O(1) 查找)
  • 预分配哈希表替代线性数组
  • 利用 producer_id 递增特性，只检查最近 N 个
2. TensorMap 链遍历优化
  • 增大 PTO2_TENSORMAP_NUM_BUCKETS (65536 → 262144)
  • 更激进的链截断策略
  • 考虑开放寻址法替代链地址法
  • LRU 缓存热点条目
3. 锁竞争优化
  • 无锁 fanout 队列 (CAS 链表)
  • 批量添加消费者，减少锁获取次数
  • 每任务 fanout 队列独立锁
  • 读写分离：读操作无锁
4. 原子操作优化
  • 批量更新 refcount
  • 使用 relaxed 内存序减少开销
  • thread-local 计数器 + 定期合并
  • 合并多个状态到单个原子变量
5. 内存分配优化
  • 预分配输出缓冲区池
  • 动态调整任务窗口大小
  • 异步通知替代自旋等待
  • 分层内存池 (小/中/大)
6. 清理策略优化
  • 增量清理：每次只清理一部分
  • 后台线程异步清理
  • 提高清理阈值，减少触发频率
  • 基于负载自适应调整
